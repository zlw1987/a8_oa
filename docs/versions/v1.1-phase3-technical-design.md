# V1.1 Phase 3 Technical Design

## Scope

V1.1 Phase 3 focuses on Remaining Go-Live Control Closure.

Included:

1. System Setup / Currency Setup Hardening.
2. Department General Budget Handling.
3. Approval Rule Snapshot / Versioning.
4. Budget Adjustment Approval.
5. Duplicate Reimbursement / Invoice Detection.
6. Attachment Retention / Deletion Control.
7. Optional report drill-down.

Excluded:

- AI receipt OCR.
- Full mobile app.
- Vendor onboarding.
- PO / invoice / 3-way match.
- ACH reimbursement payment.
- GL / ERP integration.
- Advanced BI dashboard.
- Large UI redesign.

## Design Principles

- Do not reopen Phase 1 or Phase 2 workflows unless there is a regression.
- Keep Project as the budget container for this phase.
- Preserve historical approval context at submission time.
- Manual budget changes must flow through approval, not silent ledger mutation.
- Duplicate detection should warn and route to accounting review; it should not delete records.
- Attachments used as audit evidence should not disappear after posting or period close.
- Keep Phase 3 UI operational and table/form based; avoid dashboard-heavy redesign.

## Existing Foundations

Already available:

- Project budget ledger with `RESERVE`, `CONSUME`, `RELEASE`, and `ADJUST`.
- PR/TR approval task instances.
- Accounting Review Item framework.
- Company card duplicate review.
- Line-level actual expense attachments.
- Accounting Period close/locking.
- System Setup landing page.
- Finance Reports with base-currency formatting and CSV export.

## Phase 3A: System Setup / Currency Setup Hardening

### Goal

System Setup should be a real business setup hub. Finance/Admin users should not need to use Django Admin for normal currency, exchange-rate, or FX variance policy setup.

### Required Behavior

- System Setup clearly shows company base currency.
- Active currencies can be maintained from business UI.
- Exchange rates can be maintained from business UI.
- FX Variance Policies can be maintained from business UI.
- System Setup links to business setup pages instead of Django Admin where a business page exists.
- System Setup should surface readiness warnings such as missing active currencies or missing department general budget setup.

### Base Currency Rule

Company base currency remains `USD` for this phase.

Reason:

- Budget ledger and finance reports are currently base-currency controlled.
- Switching base currency dynamically would require a larger conversion and migration strategy.

Phase 3 should expose base currency clearly, not make it freely editable.

### UI

Finance setup pages:

- Currencies.
- Exchange Rates.
- FX Variance Policies.

System Setup:

- Setup Health section.
- User/Permission setup links.
- Approval/Finance Policy links.
- Currency/Exchange Rate links.
- System Health links.

### Tests

- Finance/Admin can open currency list/create/edit pages.
- Finance/Admin can open exchange rate list/create/edit pages.
- Finance/Admin can open FX variance policy list/create/edit pages.
- Normal requester cannot access these setup pages.
- System Setup links use business setup pages instead of Django Admin for currency/FX/rates.

## Phase 3B: Department General Budget Handling

### Goal

Department recurring general spending should use controlled annual department general projects instead of relying on manual project naming.

### Recommended Model

Add `DepartmentGeneralProject` or equivalent setup model.

Suggested fields:

- `department`
- `fiscal_year`
- `project`
- `budget_amount`
- `is_active`
- `created_by`
- `created_at`
- `updated_at`

Constraints:

- One active general project per department per fiscal year.
- The linked project should belong to the same department.

### Behavior

- Finance/Admin can define annual department general projects.
- General PR/TR should default or restrict to the current department general project where the request is marked as general department spend.
- Users should not select another department's general project for general spending.
- Finance Reports should identify department general spending.
- System Setup should show missing setup warnings for departments without current fiscal-year general projects.

### UI

System Setup:

- Department General Budget Setup shortcut.
- Setup health card: missing current-year general projects.

Setup page:

- Department.
- Fiscal Year.
- General Project.
- Budget Amount.
- Active.
- Missing Setup Warning.

### Tests

- Department has active general project for fiscal year.
- Duplicate active general project for same department/year is blocked.
- General PR/TR links to correct department general project.
- User cannot select another department's general project where restricted.
- System Setup warns when department general project is missing.

## Phase 3C: Approval Rule Snapshot / Versioning

### Goal

Historical approvals must remain understandable after approval rules change.

### Existing Confirmation

`ApprovalTask` already preserves some task-instance context:

- Rule FK.
- Step FK.
- Step number.
- Step name.
- Assigned user.
- Candidate rows.
- Task creation/action timestamps.
- Approval history rows.

However, if the linked rule/step is edited later, code/name/type context can still be reinterpreted through current rule data. Phase 3 should snapshot enough display/audit fields at task creation time.

### Recommended Fields

On `ApprovalTask`:

- `approval_rule_code_snapshot`
- `approval_rule_name_snapshot`
- `approval_rule_version_snapshot`
- `step_name_snapshot`
- `step_type_snapshot`
- `assigned_user_snapshot`
- `candidate_pool_snapshot`
- `snapshot_created_at`

Alternative:

- Add `ApprovalWorkflowSnapshot` model linked to request content type/object and individual task rows.

Recommended first implementation:

- Add task-level snapshot fields because approval tasks already represent submitted workflow instances.

### Behavior

- When PR/TR/amendment approval tasks are created, copy rule and step context into snapshot fields.
- Changing approval rules later must not alter existing snapshot values.
- New requests use the latest rule/snapshot values.
- PR/TR detail Approval Workflow section should show the rule code/name/version used.

### Tests

- Submit request under approval rule version A.
- Modify approval rule to version B.
- Existing request still shows version A snapshot.
- New request uses version B snapshot.
- Historical approval history remains stable.

## Phase 3D: Budget Adjustment Approval

### Goal

Manual budget adjustment must not be posted silently.

### Recommended Model

Add `BudgetAdjustmentRequest`.

Suggested fields:

- `project`
- `adjustment_amount`
- `adjustment_type`
- `reason`
- `requested_by`
- `submitted_at`
- `approved_by`
- `approved_at`
- `posted_by`
- `posted_at`
- `status`
- `ledger_entry`

Statuses:

- `DRAFT`
- `SUBMITTED`
- `APPROVED`
- `REJECTED`
- `POSTED`
- `CANCELLED`

### Behavior

- Draft/submitted adjustment does not affect budget.
- Approval is required before posting.
- Approved adjustment creates `ProjectBudgetEntry` with entry type `ADJUST`.
- Rejected/cancelled adjustment does not affect budget.
- All adjustments require reason.
- Unauthorized users cannot approve or post adjustments.

### UI

Project detail:

- Budget Adjustment Requests section.
- Create Adjustment Request action for Finance/Admin.
- Adjustment history linked to budget ledger.

Finance/System Setup:

- Budget Adjustment Queue.

### Tests

- Draft adjustment does not affect budget.
- Submitted adjustment waits for approval.
- Approved adjustment creates ADJUST ledger entry.
- Rejected adjustment does not affect budget.
- Unauthorized user cannot approve/post adjustment.

## Phase 3E: Duplicate Reimbursement / Invoice Detection

### Goal

Detect possible duplicate actual expenses and invoices, not just duplicate company card transactions.

### Matching Fields

Use a conservative candidate score from:

- Requester.
- Vendor / merchant.
- Amount.
- Expense date.
- Invoice number.
- Reference number.
- Receipt file hash.
- Attachment file hash.
- Payment method.

### Recommended Model Changes

Add optional hash/reference fields where needed:

- Actual expense line receipt hash or linked attachment hash.
- Normalized vendor/reference helpers.

Accounting Review Item can continue to carry duplicate review items.

### Behavior

- When actual expense is recorded, search for candidate duplicates.
- Same vendor/date/amount/reference should create duplicate review.
- Same invoice number should create duplicate review.
- Same receipt file hash should create duplicate review when available.
- Do not automatically delete or block legitimate transactions unless a future policy says `BLOCK`.
- Accounting Review Detail should link source and duplicate candidates.

### Tests

- Same vendor/date/amount/reference creates duplicate review.
- Same invoice number creates duplicate review.
- Same receipt file hash creates duplicate review.
- Accounting can resolve duplicate review.
- Legitimate duplicate transactions are not automatically deleted.

## Phase 3F: Attachment Retention / Deletion Control

### Goal

Receipt/invoice evidence should not be freely deleted after approval, actual posting, or period close.

### Recommended Model Changes

Prefer soft delete / void metadata:

- `is_deleted`
- `deleted_by`
- `deleted_at`
- `delete_reason`
- `replaced_by`

If physical files remain in storage, UI should hide deleted files by default but show audit history.

### Rules

Before submission:

- Requester can delete own attachment.

After submission:

- Requester deletion is restricted or audited.

After approval / actual posting:

- Only Accounting/Admin can delete or replace support files.
- Reason required.

After period close:

- Normal deletion is blocked.
- Replacement/void should require controlled adjustment or admin override with reason.

### UI

Request detail attachments:

- Delete/Replace action with disabled reason.
- Replacement reason form.
- Attachment history section.

### Tests

- Draft attachment can be deleted by owner.
- Posted receipt cannot be deleted by requester.
- Accounting replacement requires reason.
- Closed-period attachment deletion is blocked.
- Attachment deletion/replacement is audited.

## Optional Phase 3G: Report Drill-Down

Keep this low-risk and link-based.

Suggested drill-downs:

- Project Budget Summary -> Project Budget Ledger.
- Over-Budget Exception -> Accounting Review Detail.
- Unmatched Card -> Card Transaction Detail.
- Department Spending -> filtered project/request list.
- Open Reserve -> source PR/TR.

Do not build advanced BI dashboards in Phase 3.

## Permissions

Department general budget:

- Finance/Admin can configure.
- Normal users can only use visible/allowed department general project options.

Approval snapshot:

- Visible to request participants and finance/audit users through request detail.
- Setup editing remains restricted.

Budget adjustment:

- Request/post/approve should be separated by role.
- Unauthorized users cannot approve/post adjustment.

Duplicate reimbursement:

- Accounting can resolve/approve exception.
- Requester can view own duplicate review status where relevant.

Attachment retention:

- Requester can delete only safe draft attachments.
- Accounting/Admin replacement requires reason.
- Closed-period deletion is blocked.

## Phase 3 Exit Criteria

Phase 3 is complete only if:

- Department general budget setup exists and missing setup is visible.
- Approval rule snapshots preserve historical approval context after rule edits.
- Manual budget adjustments require approval before ledger posting.
- Duplicate actual expense/invoice detection creates review items without deleting legitimate records.
- Attachment retention/deletion controls protect posted and closed-period evidence.
- Required tests pass.
- Finance/accounting validation confirms go-live control closure.
