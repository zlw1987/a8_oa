# V1.1 Phase 2 Technical Design

## Scope

V1.1 Phase 2 focuses on Expense Control And Approval Operations.

Included:

1. Line-level receipt matching UI.
2. Direct Project Cost Policy UI and workflow.
3. Approval Delegation UI.
4. Approval escalation / overdue handling.

Excluded:

- AI receipt OCR.
- Full mobile app.
- Vendor onboarding.
- PO / invoice / 3-way match.
- ACH reimbursement payment.
- GL / ERP integration.
- Advanced BI dashboard.

## Design Principles

- Do not reopen Phase 1 financial integrity workflows unless there is a regression.
- Preserve request-level attachments for quotes and general support.
- Receipt/invoice support should be linked to the exact actual expense line when used for accounting evidence.
- Direct project cost must not become a backdoor around PR/TR approval.
- Delegation must never create requester self-approval.
- Overdue visibility should be simple first; avoid a large workflow engine in this phase.
- Templates should render prepared UI data; business rules should stay in services/presentation helpers.

## Phase 2A: Line-Level Receipt Matching UI

### Goal

Accounting should know exactly which actual expense line has receipt or invoice support.

Request-level attachments remain valid for quotes, contracts, and general support, but should not automatically satisfy every actual expense line.

### UI Changes

PR detail actual spend table should show:

- Entry type.
- Spend date.
- Base amount.
- Original transaction amount.
- Receipt status.
- Invoice status.
- Linked attachments.
- Missing receipt review link.
- Actions: Upload Receipt, Link Existing Attachment.

TR detail actual expense table should show the same evidence columns for each actual expense line.

### Forms

Purchase:

- `PurchaseActualExpenseAttachmentUploadForm`
- `PurchaseActualExpenseAttachmentLinkForm`

Travel:

- `TravelActualExpenseAttachmentUploadForm`
- `TravelActualExpenseAttachmentLinkForm`

Fields:

- Attachment type: Receipt / Invoice / Quote / Other.
- File upload for direct upload.
- Existing request attachment for link-existing flow.

### URLs

Purchase:

- `purchase:pr_actual_attachment_upload`
- `purchase:pr_actual_attachment_link`

Travel:

- `travel:tr_actual_attachment_upload`
- `travel:tr_actual_attachment_link`

### Service Logic

Add helpers in `finance.services`:

- `actual_expense_has_attachment(actual_expense, attachment_type)`
- `build_actual_expense_evidence_status(actual_expense, request_obj)`
- `resolve_missing_receipt_review_for_actual(actual_expense, acting_user=None)`

Receipt policy evaluation should check line-level links first.

Initial rule:

- If a receipt/invoice is required and the specific actual line has a matching linked attachment, no missing receipt review is created.
- If a missing receipt review already exists and a matching line-level attachment is linked later, accounting can resolve it manually in Phase 2.
- Optional automatic resolution can be added only if it is safe and clearly audited.

### Accounting Review Detail

For missing receipt review items, show:

- Source request.
- Actual expense line/date.
- Amount.
- Required evidence type.
- Current linked attachments.
- Direct link back to source PR/TR detail.

### Closeout

Closeout remains blocked by unresolved AccountingReviewItem records.

Line-level receipt status should prevent false pass scenarios by making missing evidence review line-specific. Request-level attachments should not satisfy every actual line automatically.

### Tests

Required:

- Actual expense requires receipt, no line attachment linked -> missing receipt review.
- Upload receipt to actual expense line -> receipt status shows linked.
- Link existing request attachment to actual expense -> receipt status shows linked.
- Request-level attachment alone does not satisfy all actual lines.
- Accounting Review Detail shows exact missing actual expense line.

## Phase 2B: Direct Project Cost Policy UI And Workflow

### Goal

Direct project cost allocation from company card transactions must be policy-controlled and visible.

### Setup UI

New pages:

- Direct Project Cost Policy list.
- Direct Project Cost Policy create/edit form.

Routes:

- `finance:direct_project_cost_policy_list`
- `finance:direct_project_cost_policy_create`
- `finance:direct_project_cost_policy_edit`

Form fields:

- policy code.
- policy name.
- department.
- project.
- project type.
- payment method.
- amount from.
- amount to.
- currency.
- action.
- requires receipt.
- requires project owner review.
- priority.
- active.

Help text:

- Lower priority number matches first.
- BLOCK prevents direct project cost allocation.
- REVIEW creates accounting review.
- REQUIRE_PROJECT_OWNER_APPROVAL creates/marks owner review requirement.

### Card Allocation Behavior

Existing `allocate_card_transaction` should remain the policy execution point.

Required behavior:

- ALLOW: allocation succeeds.
- REVIEW: allocation succeeds and creates review item.
- REQUIRE_PROJECT_OWNER_APPROVAL: allocation succeeds and sets project owner review status / creates review item.
- BLOCK: allocation is rejected.

### UI Display

Card Transaction Detail allocations table should show:

- Direct project cost policy.
- Direct project cost action.
- Project owner review status.
- Related review item link.

Finance Reports should include direct project cost in project consumed spend through existing budget ledger entries.

### Tests

Required:

- Policy ALLOW -> direct project allocation succeeds.
- Policy REVIEW -> accounting review item is created.
- Policy REQUIRE_PROJECT_OWNER_APPROVAL -> project owner review status is pending.
- Policy BLOCK -> allocation is blocked.
- Unauthorized user cannot allocate card transaction directly to project.

## Phase 2C: Approval Delegation UI

### Goal

Approvers should be able to delegate approval authority for a date range, and delegated approvals should remain auditable.

### Screens

New pages:

- My Delegations.
- Create Delegation.
- Edit/Deactivate Delegation.
- Active Delegations, admin view.
- Admin Reassign Approval Task.

Routes:

- `approvals:my_delegations`
- `approvals:delegation_create`
- `approvals:delegation_edit`
- `approvals:active_delegations`
- `approvals:task_reassign`

### Delegation Rules

- Original approver can delegate to another active user.
- Start date and end date are required.
- Delegation applies only during active date range.
- Optional filters: department and request type.
- Delegate cannot be the requester for the task.
- Delegation does not bypass existing requester self-approval prevention.

### Service Logic

Add helpers in `approvals.services` or `approvals.access`:

- `get_active_delegations_for_user(user, date=None)`
- `user_can_act_via_delegation(user, task)`
- `get_effective_approver_context(user, task)`
- `reassign_approval_task(task, new_user, acting_user, reason)`

Approval permission checks should consider active delegation after direct assignment/pool rules.

### History

Approval history should show:

- Original approver.
- Delegate user.
- Acting user.
- Delegation reference.
- Comment.

If existing history fields are not enough, add a concise comment convention first:

```text
Approved by delegate user_a on behalf of original approver user_b.
```

Avoid schema churn unless needed for audit clarity.

### Tests

Required:

- Active delegation lets delegate approve.
- Expired delegation does not let delegate approve.
- Delegation to requester is blocked at action time.
- Admin reassignment requires reason.
- Approval history records delegation context.

## Phase 2D: Approval Escalation / Overdue Handling

### Goal

Make overdue approval work visible and prepare simple escalation execution without overbuilding.

### Minimum UI

- Dashboard overdue approval card should remain visible for approvers/managers.
- My Tasks should highlight overdue tasks.
- Approval task list/filter should support overdue filter.
- Admin/manager should be able to see overdue approval tasks.

### Optional Management Command

Command:

```powershell
python manage.py process_approval_escalations
```

Initial behavior:

- Find overdue active approval tasks.
- Evaluate `ApprovalEscalationPolicy`.
- Write history/comment or notification log.
- Optionally reassign only when policy has explicit target user.

Avoid automatic broad reassignment until accounting/management validates rules.

### Tests

Required:

- Overdue tasks are highlighted.
- Overdue filter returns expected tasks.
- Escalation command finds overdue tasks.
- Escalation does not allow requester self-approval.
- Existing approval tests still pass.

## Delivery Order

1. Line-level receipt matching UI.
2. Direct Project Cost Policy setup pages.
3. Direct Project Cost policy display on card allocation.
4. Approval Delegation UI.
5. Overdue approval task visibility.
6. Optional escalation command.
7. Regression tests.
8. Accounting / manager validation.

## Permissions

Line-level receipt:

- Requester can upload/link support only where they can manage request attachments.
- Accounting can upload/link support while resolving evidence issues.

Direct project cost policy:

- Finance Admin can configure policies.
- Accounting can allocate card transactions.
- Unauthorized users cannot allocate directly to project.

Delegation:

- Approvers can manage their own delegation.
- Admin/finance/system users can view active delegations and reassign stuck tasks where permitted.

## Phase 2 Exit Criteria

Phase 2 is complete only if:

- Actual expense lines show receipt/invoice status.
- Missing receipt review points to exact actual expense line.
- Direct Project Cost Policy setup exists.
- Direct project card allocation applies policy and shows result.
- Delegation UI exists and prevents requester self-approval.
- Overdue approval tasks are visible.
- Required tests pass.
- Accounting/manager UAT validates the workflow.
