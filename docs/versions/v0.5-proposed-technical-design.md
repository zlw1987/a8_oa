# V0.5 Proposed Technical Design

## Goal

V0.5 should close V0.4 business-control gaps before unrelated feature expansion.

Recommended scope:

1. Request amendment / revision flow.
2. Receipt and attachment policy.
3. Accounting workbench improvements.
4. Finance and budget reporting.
5. Production readiness.

## Priority 1: Request Amendment / Revision Flow

### Phase 1 Implementation

Implemented as linked supplemental/amendment PR/TR records using each request model's existing `parent_request` relationship.

Delivered behavior:

- Approved original requests remain immutable through the existing edit permission rules.
- A supplemental/amendment request links back to the original request.
- Amendment detail pages show original approved amount, amendment delta, revised amount, and reason.
- Original request detail pages list linked amendment requests.
- Amendment requests run their own approval workflow.
- Amendment requests do not reserve additional budget on submit.
- Additional budget is reserved only after amendment approval.
- Original request closeout is blocked while linked amendment requests are still open.
- Existing accounting review resolution still controls whether an over-budget actual can close.

Current intentional scope boundary:

- Amendment is amount-level and request-level in Phase 1.
- Line-level diff visualization is not implemented yet.
- Automatic accounting review resolution after amendment approval is not implemented; accounting still makes the final review decision.

### Business Rules

- Approved request should not be directly edited.
- If the approved amount needs to increase, requester creates an amendment or revision.
- Original approved request remains unchanged.
- Revised amount requires approval.
- Additional budget is reserved only after amendment approval.
- Amendment history links back to the original request.
- Closeout is blocked while amendment-required actuals are unresolved.

### Proposed Model Changes

Option A: Generic amendment model.

- `RequestAmendment`
  - `source_type`
  - `source_content_type`
  - `source_object_id`
  - `source_object`
  - `amendment_no`
  - `original_amount`
  - `revised_amount`
  - `delta_amount`
  - `reason`
  - `status`
  - `created_by`
  - `submitted_at`
  - `approved_at`
  - `approved_by`
  - `rejected_at`
  - `rejected_by`

Option B: Separate amendment models.

- `PurchaseRequestAmendment`
- `TravelRequestAmendment`

Preferred approach:

- Use separate amendment models if purchase and travel line-level amendments diverge.
- Use a generic header amendment if V0.5 only needs amount-level revisions.

### Proposed Approval Integration

- Add amendment request type values or source subtype to approval rule matching.
- Create approval tasks against the amendment object.
- Approval completion updates amendment status.
- On amendment approval, create budget reserve entry for the delta only.
- Original request remains immutable.

### Proposed Closeout Rules

- If actual policy action is `AMENDMENT_REQUIRED`, closeout remains blocked until:
  - an amendment exists,
  - the amendment is approved,
  - accounting review item is resolved,
  - any additional reserve/consume entries are balanced.

## Priority 2: Receipt / Attachment Policy

### Business Rules

- Amount below 25: receipt optional.
- Amount 25 or above: receipt required.
- Amount 500 or above: invoice required.
- Company card transaction: receipt required unless exception approved.

### Proposed Model Changes

- `ReceiptPolicy`
  - `policy_code`
  - `request_type`
  - `expense_type`
  - `payment_method`
  - `amount_from`
  - `amount_to`
  - `requires_receipt`
  - `requires_invoice`
  - `allows_exception`
  - `priority`
  - `is_active`

- Attachment audit model or generic content audit extension:
  - attachment added,
  - attachment deleted,
  - user,
  - timestamp,
  - source request,
  - file metadata.

### Proposed Behavior

- Evaluate receipt policy during actual spend / actual expense entry.
- Missing required receipt creates accounting review item with reason `MISSING_RECEIPT`.
- Closeout blocked while required receipt review remains unresolved.
- Attachment add/delete creates audit entries.

## Priority 3: Accounting Workbench Improvement

### Required Filters

- pending review
- over-budget
- missing receipt
- card transaction issue
- duplicate suspected
- amendment required
- requester
- department
- project
- aging days

### Required Columns

- source request
- actual expense
- policy result
- reason
- amount variance
- required action
- reviewer decision
- comments
- aging days

### Proposed Model Changes

Likely no major model change is required if `AccountingReviewItem` remains the central queue.

Potential additions:

- `required_action`
- `severity`
- `due_at`
- `resolved_reason`
- `related_amendment`

## Priority 4: Finance / Budget Reporting

### Minimum Reports

- Project budget summary.
- Department spending summary.
- Reserved vs consumed budget.
- Open requests with remaining reserve.
- Over-budget exception report.
- Company card unmatched transaction report.
- Accounting review aging report.

### Proposed Implementation

- Use simple Django list/table views first.
- Reuse existing project budget entries and accounting review items.
- Add export later only if business users request it.

### Possible Query Helpers

- `projects.reporting.get_project_budget_summary`
- `finance.reporting.get_over_budget_exceptions`
- `finance.reporting.get_card_unmatched_transactions`
- `finance.reporting.get_accounting_review_aging`
