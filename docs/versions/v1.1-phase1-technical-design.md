# V1.1 Phase 1 Technical Design

## Scope

Phase 1 turns the existing financial-integrity foundation into complete operational workflows:

- Period Close / Month-End Close UI.
- Refund / Credit / Reversal workflow.
- Reopen / Correction workflow.

Implementation status: completed for the first operational slice. The system now includes Accounting Period list/detail/create/close/reopen UI, a period close checklist helper, PR/TR refund-credit-reversal forms, and PR/TR reopen-for-correction forms. Closed periods block ordinary actual expense posting, card transaction creation/allocation, and refund/credit entry.

This phase should not add unrelated product features. It should make financial records safer, more auditable, and easier for finance users to operate after go-live.

## Design Principles

- Use USD base currency for budget ledger and finance control.
- Never delete original financial records to correct history.
- Use reversal, refund, credit, or adjustment entries instead of overwriting posted records.
- Closed accounting periods block ordinary financial changes.
- Post-close corrections require explicit Finance/Admin authority, reason, and audit trail.
- PR and TR should share the same operating pattern wherever possible.
- UI should show disabled action reasons instead of hiding actions silently.

## Phase 1A: Period Close / Month-End Close UI

### URLs

Add routes under `finance/`:

- `finance:accounting_period_list`
- `finance:accounting_period_create`
- `finance:accounting_period_detail`
- `finance:accounting_period_close`
- `finance:accounting_period_reopen`

### Views

Add views in `finance/views.py`:

- `accounting_period_list`
- `accounting_period_create`
- `accounting_period_detail`
- `accounting_period_close`
- `accounting_period_reopen`

Permission:

- List/detail: accounting manager, finance admin, system admin.
- Create/close/reopen: finance admin or explicitly authorized accounting manager.
- Normal requesters and approvers must not access period close UI.

Use shared permission helpers in `common.permissions`; do not add broad `is_staff` checks.

### Forms

Add:

- `AccountingPeriodForm`
- `AccountingPeriodCloseForm`
- `AccountingPeriodReopenForm`

Close/reopen forms must require notes/reason.

### Period Close Checklist

The detail page should show a checklist before close:

- No pending accounting review items in period.
- No unmatched or partially matched card transactions in period.
- No missing receipt review items in period.
- No open PR/TR with remaining reserve dated inside the period.
- No open correction workflow in period.

Initial implementation can compute warnings and allow Finance/Admin to close with notes. Later hardening can make selected checklist items blocking.

### Close Behavior

Closing a period should:

- Set `status = CLOSED`.
- Set `closed_by`.
- Set `closed_at`.
- Save notes.
- Block ordinary financial changes through `enforce_accounting_period_open`.

### Reopen Behavior

Reopen should be admin-only.

Reopening should:

- Require reason.
- Change status back to `OPEN` or `CLOSING`.
- Preserve original closed_by / closed_at history until a future audit model is added.
- Write an audit/history entry if an accounting-period history model is added in a later iteration.

### Templates

Add:

- `finance/templates/finance/accounting_period_list.html`
- `finance/templates/finance/accounting_period_detail.html`
- `finance/templates/finance/accounting_period_form.html`

System Setup should link to Accounting Periods.

### Acceptance Criteria

- Finance/Admin can create accounting periods.
- Finance/Admin can view current open period.
- Period detail shows close checklist.
- Closed period blocks actual expense posting and card allocation changes.
- Close and reopen require notes/reason.
- Normal requesters cannot access period close pages.

## Phase 1B: Refund / Credit / Reversal Workflow

### URLs

Add routes:

- `purchase:pr_record_refund`
- `travel:tr_record_refund`

Optional later:

- actual expense detail page for PR actual spend.
- actual expense detail page for TR actual expense.

### Forms

Add:

- `PurchaseRefundForm`
- `TravelRefundForm`

Fields:

- Original actual expense, optional but preferred.
- Refund date.
- Entry type: REFUND / CREDIT_MEMO / REVERSAL / ADJUSTMENT.
- Amount, entered as positive number in UI.
- Vendor / merchant.
- Reference no.
- Notes / reason.

The service/model method should store the ledger effect as a negative base amount.

### UI

On PR/TR detail actual expense sections:

- Add `Record Refund / Credit` action for authorized accounting/finance users.
- Show actual rows with:
  - Entry Type.
  - Base Amount.
  - Original Transaction Amount.
  - Linked Original Expense.
  - Net Actual Spend.

For closed requests:

- Show action disabled unless correction workflow is open or Finance/Admin has explicit adjustment path.

For closed accounting periods:

- Disable ordinary refund.
- Explain that post-close adjustment is required.

### Budget Ledger

Refund / credit should:

- Create a separate negative actual record.
- Create a negative `CONSUME` budget ledger entry.
- Never delete or mutate original actual expense.

### Acceptance Criteria

- Refund reduces consumed budget.
- Original spend remains visible.
- Net actual spend displays correctly.
- Refund after period close is blocked unless post-close adjustment flow is used.
- Refund action is permission-aware.

## Phase 1C: Reopen / Correction Workflow

### URLs

Add routes:

- `purchase:pr_reopen_correction`
- `travel:tr_reopen_correction`

Existing close routes should be reused for reclose after correction.

### Forms

Add:

- `PurchaseReopenCorrectionForm`
- `TravelReopenCorrectionForm`

Fields:

- Reason, required.
- Correction reference, optional.

### UI

On closed PR/TR detail pages:

- Show `Reopen For Correction` action for authorized Finance/Admin users.
- If period is closed, show disabled reason: use adjustment workflow instead.
- Show Correction History panel:
  - Reopened by.
  - Reopened at.
  - Reason.
  - Correction reference.
  - Reclosed by.
  - Reclosed at.
  - Correction status.

When reopened:

- Request returns to approved / expense-pending stage.
- Existing ledger and actual expense records stay intact.
- Finance can add refund/credit/correction entries.
- Reclose runs existing closeout validation.

### Acceptance Criteria

- Requester cannot reopen a closed request.
- Finance/Admin can reopen closed PR/TR with reason.
- Reopen writes request history.
- Existing ledger entries are not deleted.
- Reclose uses the existing closeout validation.
- Closed-period requests cannot be directly reopened for normal correction.

## Reporting Impact

Finance Reports should include refund and credit entries in consumed totals because they are negative consume ledger entries.

Future drill-down should show:

- Gross actual spend.
- Refund / credit total.
- Net actual spend.

Phase 1 can start with request detail net actual display and ledger correctness.

## Audit Requirements

Minimum Phase 1 audit:

- PR/TR request history records reopen and refund actions.
- Period close stores closed_by, closed_at, and notes.
- Refund/credit actual records preserve created_by and created_at.
- Budget ledger keeps negative consume entries instead of mutation.

Future hardening:

- Dedicated AccountingPeriodHistory.
- Dedicated CorrectionHistory model.
- Post-close adjustment approval.

## Tests

### Unit / Service Tests

- Closed period blocks PR actual spend.
- Closed period blocks TR actual expense.
- Closed period blocks card allocation.
- Refund creates negative actual record.
- Refund creates negative budget consume entry.
- Reopen requires Finance/Admin permission.
- Reopen requires reason.

### Permission Tests

- Requester cannot access accounting period UI.
- Accounting/Finance role can view period list.
- Finance/Admin can close period.
- Requester cannot reopen closed PR/TR.

### Template Tests

- Period list renders.
- Period detail checklist renders.
- Closed PR/TR shows reopen action only for authorized users.
- Refund action appears for accounting/finance users.
- Disabled reasons render for closed period.

### Regression Tests

- Existing PR/TR closeout tests still pass.
- Accounting review blockers still block closeout.
- Company card allocation over-allocation protection still works.
- Finance Reports still show base currency amounts.

## Implementation Order

1. Period close forms/views/templates.

3. PR/TR refund forms/views/templates.
4. PR/TR correction reopen forms/views/templates.
5. Request detail panels for correction history and net actual spend.
6. Tests.
7. User guide updates.

## Out Of Scope For Phase 1

- Excel export.
- Line-level receipt matching UI.
- Direct project cost policy UI.
- Approval delegation UI.
- Exchange rate import UI.
- Notification/reminder automation.
- OCR, AP/PO, GL integration, advanced dashboards.
