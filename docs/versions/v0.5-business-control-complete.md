# V0.5 Business Control Completion

## Scope

V0.5 completes the stabilization work identified during V0.4 UAT and closes the remaining finance-control gaps before adding unrelated features.

Implemented scope:

- Request amendment / revision baseline for approved PR and TR increases.
- Receipt and invoice policy setup.
- Missing receipt accounting review item creation.
- Company card allocation guardrails.
- Duplicate card transaction review workflow.
- Accounting review workbench filters.
- Basic finance and budget reports.
- Production readiness checklist and finance default seeding command.

## Amendment / Revision Flow

Approved requests are not directly edited for budget increases. Users create supplemental PR or TR records linked to the original request.

Implemented behavior:

- Original request remains linked and visible.
- Supplemental request carries only the additional amount.
- Approval workflow runs on the supplemental request.
- Additional reserve is created only after supplemental approval.
- Parent closeout is blocked while supplemental requests remain open.
- AMENDMENT_REQUIRED actual-expense policy creates an accounting review item and blocks closeout until resolved.

Known limitation:

- V0.5 supports supplemental increase requests as the operational amendment path. A richer compare screen for every changed field can be added later if business users need formal redline-style amendment review.

## Receipt / Attachment Policy

Finance can configure receipt policies by:

- Request type.
- Department.
- Project type.
- Expense type.
- Payment method.
- Currency.
- Amount range.
- Receipt required.
- Invoice required.
- Exception allowed.
- Priority.

Implemented behavior:

- Actual spend and travel actual expense evaluate receipt policy after posting.
- Company card allocations evaluate receipt policy as company card spend.
- Missing required receipt or invoice creates a MISSING_RECEIPT accounting review item.
- Request closeout is blocked while the review item is unresolved.
- PR and TR attachment upload/delete actions are written to content audit history.

Current matching rule:

- V0.5 checks request-level non-accounting attachments.
- Line-level receipt-to-expense matching is not implemented yet.

## Accounting Workbench

The Accounting Review Queue now supports business filters for:

- Status.
- Reason.
- Source type.
- Policy action.
- Requester.
- Department.
- Project.
- Minimum aging days.
- Keyword.

Each row shows:

- Source request or card transaction.
- Requester, department, project.
- Reason and policy action.
- Amount and variance.
- Aging days.
- Required action guidance.
- Reviewer decision and comments.

## Company Card Controls

Implemented behavior:

- Allocation amount cannot exceed remaining unallocated transaction amount.
- Split allocation is supported through multiple allocations.
- Match status updates to unmatched, partially matched, or matched.
- Fully matched transactions can be marked reviewed only when unresolved review items do not exist.
- Duplicate candidate transactions create DUPLICATE_CARD accounting review items.
- Duplicate detection does not automatically block legitimate transactions.

## Finance Reports

Implemented simple table reports:

- Project budget summary.
- Department spending summary.
- Reserved vs consumed budget.
- Open requests with remaining reserve.
- Over-budget exception report.
- Company card unmatched transaction report.
- Accounting review aging report.

These are intentionally table-first reports for production readiness, not advanced dashboards.

## Default Setup

Run:

```bash
python manage.py seed_finance_defaults
```

The command seeds:

- Over-budget warning, review, amendment-required, and block policies.
- Receipt optional below 25.
- Receipt required from 25.
- Invoice required from 500.
- Company card receipt required.

## Known Limitations

- Receipt policy checks request-level attachments, not line-level receipt links.
- Duplicate card detection uses same date, merchant, amount, and reference; fuzzy matching can be added later.
- Reports are simple operational tables without export or scheduled delivery.
- Accounting workbench is staff-permission based; finer-grained finance role permissions can be added when roles are finalized.
