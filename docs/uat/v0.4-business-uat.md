# V0.4 Business UAT Checklist

## Purpose

V0.4 UAT verifies the business workflows from an end-user perspective before V0.5 feature development begins.

The goal is to confirm that purchase, travel, finance policy, budget ledger, and company card controls are understandable, correct, and stable for business users.

## Test Data Setup

Prepare these records before UAT:

- One requester user.
- One manager or approver user.
- One accounting or finance reviewer user.
- One active department with manager assigned.
- One active project with enough budget for normal cases.
- Approval rules for purchase and travel.
- Finance over-budget policies:
  - Warning policy for approved amount 1000 and actual total 1030.
  - Review policy for approved amount 1000 and actual total 1100.
  - Amendment-required policy for approved amount 1000 and actual total 1300.
  - Block policy for approved amount 1000 and actual total 1800.
- Company card transactions for one-to-one, split allocation, and duplicate detection tests.

## A. Normal Purchase Request Flow

### Steps

1. Create a purchase request.
2. Add purchase lines.
3. Submit the request.
4. Confirm an approval rule is matched.
5. Confirm an approval task is created.
6. Approve the task as an authorized approver.
7. Confirm budget is reserved.
8. Record actual spend within the approved amount.
9. Confirm budget is consumed.
10. Close the request.
11. Confirm remaining reserve is released.

### Expected Results

- Request lifecycle is correct.
- Approval history is correct.
- Budget ledger contains reserve, consume, and release entries as expected.
- Actual spend is recorded correctly.
- Request can close cleanly.

## B. Normal Travel Request Flow

### Steps

1. Create a travel request.
2. Enter estimated travel expenses.
3. Calculate per diem if applicable.
4. Submit the request.
5. Approve the request.
6. Record actual travel expense.
7. Confirm finance policy is applied during actual-expense recording.
8. Confirm budget is consumed.
9. Close the request.

### Expected Results

- Travel estimate and actual expense are separated.
- Per diem logic works correctly.
- Finance policy is applied in the actual-expense recording path.
- Budget ledger contains the expected reserve, consume, and release entries.
- Request can close cleanly when there are no unresolved controls.

## C. Over-Budget Warning

### Steps

1. Create and approve a request with approved amount 1000.
2. Configure a policy that returns `WARNING` for actual total 1030.
3. Record actual total 1030.
4. Review the page message, request status, actual review state, and budget ledger.
5. Attempt to close the request.

### Expected Results

- Warning is shown to the user.
- Actual expense can still be recorded.
- Warning is retained on the request actual-review fields or visible audit trail.
- No unresolved accounting review item is created.
- Request can still close if no other unresolved review item exists.

### Current Behavior To Verify

- Warning sets the request actual-review state to approved-to-proceed and stores the warning message on the request review/comment fields.
- A normal actual-spend or actual-expense history entry is created.
- There is not currently a separate dedicated warning review item.

## D. Over-Budget Review

### Steps

1. Create and approve a request with approved amount 1000.
2. Configure a policy that returns `REVIEW` for actual total 1100.
3. Record actual total 1100.
4. Confirm accounting review item creation.
5. Attempt to close the request.
6. Approve or resolve the review item as a non-requester accounting reviewer.
7. Attempt closeout again.

### Expected Results

- Actual expense is recorded but not treated as fully cleared.
- Accounting review item is created.
- Request closeout is blocked while the review item is unresolved.
- Requester cannot approve or resolve their own accounting review.
- Non-requester reviewer can approve or resolve the review item.
- After resolution, closeout becomes available if no other controls are open.

## E. Amendment Required

### Steps

1. Create and approve a request with approved amount 1000.
2. Configure a policy that returns `AMENDMENT_REQUIRED` for actual total 1300.
3. Record actual total 1300.
4. Confirm whether actual expense is accepted or blocked.
5. Confirm whether accounting review item is created.
6. Confirm whether a real amendment or revision object exists.
7. Attempt to close the request before resolving the issue.
8. Resolve the accounting review item and attempt closeout again.

### Expected Results For UAT Decision

- Business users must explicitly decide whether current temporary behavior is acceptable for V0.4.
- If amendment is mandatory before posting actuals, current behavior is not sufficient and should be fixed before production.

### Current System Behavior

- Actual expense is not blocked.
- Actual expense is recorded.
- Budget consumption is created after actual recording.
- Accounting review item is created.
- Request closeout is blocked while the review item remains unresolved.
- There is no real amendment or revision workflow yet.
- Temporary handling is accounting review exception or resolution.
- Closeout is not available before review resolution.

## F. Hard Block

### Steps

1. Create and approve a request with approved amount 1000.
2. Configure a policy that returns `BLOCK` for actual total 1800.
3. Attempt to record actual total 1800.
4. Review the error message.
5. Confirm no actual spend or expense line is created.
6. Confirm no incorrect budget consumption is created.
7. Confirm whether the blocked attempt is logged.

### Expected Results

- Accounting cannot post the actual expense against the request.
- Clear error message is shown.
- No incorrect budget consumption happens.
- Attempt should be logged for production readiness.

### Current Behavior To Verify

- Actual recording is blocked by validation error.
- Actual expense line and budget consumption should not be created.
- The blocked attempt is not currently represented by a dedicated audit record.

## G. Company Card Transaction Matching

### Steps

1. Create or import a company card transaction.
2. Open the card transaction detail.
3. Allocate the transaction to a purchase request or travel request.
4. Confirm finance policy is applied.
5. Confirm request actual spend or actual travel expense is created.
6. Confirm budget consumption is created.
7. Confirm transaction match status updates.

### Expected Results

- Unmatched, partially matched, matched, and reviewed statuses are correct.
- Allocation is auditable.
- Actual spend is connected to the correct request.
- Over-budget policy is applied during allocation.

### Current Behavior To Verify

- Allocation creates request actual spend or travel actual expense with finance policy skipped in the request method because policy was already evaluated at allocation time.
- Match status is refreshed after allocation.
- There is no separate user action that marks a matched transaction as reviewed yet.

## H. Split Card Allocation

### Steps

1. Create card transaction amount 1500.
2. Allocate 1000 to PR001.
3. Allocate 500 to TR002.
4. Confirm each allocation applies policy separately.
5. Confirm total allocation and match status.
6. Attempt to allocate more than the remaining unallocated amount.

### Expected Results

- Multiple allocations are supported.
- Total allocation cannot exceed card transaction amount.
- Each allocation applies policy separately.
- Match status becomes matched after full allocation.

### Current Behavior To Verify

- Multiple allocations are supported.
- Match status is based on total allocated amount.
- A strict total-allocation ceiling was not found in the current model/service validation and must be verified during UAT.

## I. Duplicate Card Transaction Detection

### Steps

1. Create a card transaction.
2. Create a second transaction with same statement date, amount, merchant, and reference number.
3. Review the create screen and detail screen.
4. Confirm accounting can continue review.

### Expected Results

- System flags suspected duplicate.
- Accounting can review the duplicate.
- System does not automatically block legitimate transactions without user confirmation.

### Current Behavior To Verify

- Duplicate detection checks same statement date, amount, merchant name, and reference number.
- The UAT requirement says transaction date and same or similar reference should be considered; current logic should be reviewed against that business expectation.
