# V0.4 Current Behavior And Known Limitations

## AMENDMENT_REQUIRED Current Behavior

Current behavior:

- The actual expense is not blocked.
- The actual spend or actual travel expense line is recorded.
- Budget consumption is created after actual recording.
- An accounting review item is created.
- The request is marked as over estimate / pending review.
- Closeout is blocked while the accounting review item is unresolved.
- A non-requester accounting reviewer can approve or resolve the review item.
- After review resolution, closeout can continue if no other blocking controls remain.

What does not exist yet:

- No amendment request model.
- No revision number.
- No original-versus-revised amount screen.
- No separate amendment approval route.
- No additional-budget-only reservation after amendment approval.
- No immutable original approved request snapshot.

Temporary handling:

- `AMENDMENT_REQUIRED` is currently handled like a stronger accounting review exception.
- This is acceptable only if business users agree that V0.4 can stabilize without real amendment workflow.

Recommended decision:

- Treat real amendment/revision workflow as Priority 1 for V0.5.

## BLOCK Current Behavior

Current behavior:

- Actual recording raises a validation error.
- Actual spend or expense should not be created.
- Budget consumption should not be created.

Known limitation:

- There is no dedicated audit record for blocked attempts.

## WARNING Current Behavior

Current behavior:

- Actual recording is allowed.
- No accounting review item is created.
- Request actual-review fields store approved-to-proceed state and warning message.
- Normal actual-recorded history is created by the request workflow.

Known limitation:

- There is no dedicated warning audit item.

## REVIEW Current Behavior

Current behavior:

- Actual recording is allowed.
- Accounting review item is created.
- Closeout is blocked until review is resolved.
- Requester self-review is blocked.

## Company Card Current Behavior

Current behavior:

- Card transaction can be created manually.
- Allocation can link to exactly one purchase request, travel request, or project.
- Allocation to PR or TR evaluates finance policy before creating actual spend or actual travel expense.
- Request actual creation skips policy re-evaluation because policy was already applied at allocation time.
- Match status refreshes from allocated amount:
  - zero allocation: unmatched,
  - partial allocation: partially matched,
  - full or greater allocation: matched.

Known limitations:

- A strict total-allocation ceiling was not found and must be verified in UAT.
- There is no separate reviewed status transition in the current allocation workflow.
- Duplicate detection checks statement date, amount, merchant, and reference number; the business test mentions transaction date and similar reference, which may require V0.5 refinement.
