# V0.4 Business Demo Script

## Demo Goal

Show business users how V0.4 works end to end, where controls happen, and which gaps must be agreed before V0.5 starts.

## Demo Roles

- Requester: creates requests and records actuals.
- Approver: approves purchase and travel requests.
- Accounting reviewer: reviews over-budget and accounting review items.
- Finance setup user: configures policies and card transactions.

## Opening

1. Explain that V0.4 is a stabilization milestone.
2. Explain that the demo covers normal flow, exception flow, card matching, and known gaps.
3. Confirm that the purpose is UAT signoff, not new feature discovery.

## Demo 1: Normal Purchase Flow

1. Log in as requester.
2. Create purchase request with project and lines.
3. Submit request.
4. Show matched approval rule and created task.
5. Log in as approver.
6. Approve task.
7. Show purchase request status and approval history.
8. Show project budget ledger reserve entry.
9. Log in as requester or authorized actual recorder.
10. Record actual spend within approved amount.
11. Show actual spend record.
12. Show budget consume entry.
13. Close request.
14. Show remaining reserve release entry and final request status.

## Demo 2: Normal Travel Flow

1. Log in as requester.
2. Create travel request with itinerary and estimated expense lines.
3. Show per diem calculation if the test data has an active per diem policy.
4. Submit request.
5. Log in as approver and approve.
6. Show budget reserve entry.
7. Log in as requester or authorized actual recorder.
8. Record actual travel expense.
9. Show that estimated expense and actual expense are separate.
10. Show actual travel expense record.
11. Show finance policy result and budget consume entry.
12. Close request.

## Demo 3: Finance Policy Warning

1. Show active warning policy.
2. Use approved amount 1000.
3. Record actual total 1030.
4. Show warning message or saved warning state.
5. Confirm no open accounting review item exists.
6. Close request.

## Demo 4: Finance Policy Review

1. Show active review policy.
2. Use approved amount 1000.
3. Record actual total 1100.
4. Show created accounting review item.
5. Attempt closeout and show blocked message.
6. Log in as non-requester accounting reviewer.
7. Approve or resolve the review item.
8. Return to request and close successfully.

## Demo 5: Amendment Required Gap

1. Show active amendment-required policy.
2. Use approved amount 1000.
3. Record actual total 1300.
4. Show current behavior:
   - actual is recorded,
   - review item is created,
   - closeout is blocked,
   - there is no real amendment object.
5. Ask business users to confirm whether temporary accounting exception handling is acceptable for V0.4.
6. Mark amendment/revision flow as Priority 1 for V0.5.

## Demo 6: Hard Block

1. Show active block policy.
2. Use approved amount 1000.
3. Attempt actual total 1800.
4. Show validation error.
5. Confirm no actual line and no budget consumption entry were created.
6. Note whether blocked attempts require explicit audit logging before production.

## Demo 7: Company Card Matching

1. Log in as finance setup or accounting user.
2. Create card transaction.
3. Open card transaction detail.
4. Allocate to purchase or travel request.
5. Show request actual spend or actual expense created from allocation.
6. Show policy action on allocation.
7. Show budget consumption.
8. Show match status.

## Demo 8: Split Allocation

1. Create card transaction amount 1500.
2. Allocate 1000 to purchase request.
3. Allocate 500 to travel request.
4. Show both allocation rows.
5. Show final matched status.
6. Attempt excess allocation and capture current system behavior.

## Demo 9: Duplicate Detection

1. Create original card transaction.
2. Create suspected duplicate.
3. Show duplicate warning on create or detail screen.
4. Confirm that the transaction is not automatically blocked.

## Close

1. Review the UAT result matrix.
2. List failed or unclear items.
3. Decide whether issues are V0.4 stabilization fixes or V0.5 scope.
4. Confirm that V0.5 does not begin until amendment-required behavior is accepted or redesigned.
